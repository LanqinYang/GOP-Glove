# TinyML Inference Suite (Arduino)

This folder hosts Arduino sketches to benchmark and demo on-device inference for multiple gesture-recognition models. It is designed for reproducible, paper-ready latency measurements on Arduino Nano 33 BLE Sense Rev2.

## Hardware
- Board: Arduino Nano 33 BLE Sense Rev2 (nRF52840, 1 MB Flash, 256 KB RAM)
- Sensors: 5 analog inputs on A0–A4 (flex/EMG placeholders)
- Baudrate: 115200

## Models and inference backends
- 1D_CNN (TensorFlow Lite Micro, INT8)
- LightGBM (m2cgen → pure C, features on device)
- XGBoost (micromlgen → pure C, features on device)
- ADANN (pure C FP32 head; uses FPU; optional in hybrid)
- ADANN_LightGBM (fusion of ADANN + LightGBM, Arduino-friendly simplified fusion)
- Transformer_Encoder: not supported on Arduino (missing BATCH_MATMUL in current TFLM)

## Measurement protocol (fairness and paper-ready notes)
To ensure fair comparison across heterogeneous models, we provide two latency protocols:

1) Inference-only latency (default in these sketches)
- record: capture one 100×5 window and precompute the input once
  - 1D_CNN: EMA + standardization + INT8 packing → TFLM input tensor
  - LightGBM/XGBoost: EMA + per-channel features (mean/std/min/max → 20 dims)
  - ADANN/ADANN_LightGBM: prepare model-specific inputs if needed
- latency: time only the model inference call
  - 1D_CNN: `interpreter->Invoke()`
  - LightGBM/XGBoost: `predict_gesture(features_fixed)` or `clf.predict(features_fixed)`
  - ADANN: `adann_predict(x_fixed)`
- Output JSON:
  - CNN/LightGBM/ADANN: `{"latency_ms": XX.XX, "throughput_ips": YY.YY}`
  - XGBoost: `{ "latency_us": XX.XX, "throughput_ips": YY.YY }` (us granularity due to very short times)
- Rationale for papers: preprocessing (EMA, normalization, feature engineering, INT8 packing) differs across model families; we therefore isolate the inference phase for fair cross-family comparison.

2) End-to-end latency (optional)
- If needed, switch to the alternative sketches or earlier revisions that include preprocessing inside the timed loop. This is better for product-level latency but less fair across heterogeneous models.

## Usage
1) Open the desired `Latency_*/Latency_*.ino` in Arduino IDE
2) Upload to Nano 33 BLE Sense Rev2
3) Serial console (115200):
   - `record` → prepares a fixed input (precompute done once)
   - `latency [R W]` → W warmups (not recorded) + R timed runs
   - Output is a single JSON line per command for easy logging/plotting

Recommended parameters for stable results: `latency 200 10` (increase to `1000 50` for smoother averages).

## Training/export notes (how headers appear here)
- Headers are produced by training/export scripts and copied here only for Arduino mode runs. Full-mode training saves headers under `models/trained/...` without copying to Arduino folders.
- LightGBM/XGBoost use 20-dim features on device (mean/std/min/max per channel) for both training (Arduino mode) and Arduino inference to keep protocols aligned.
- ADANN uses pure C FP32 constants (leveraging M4F FPU). ADANN_LightGBM fuses a lightweight LightGBM branch with ADANN; LightGBM complexity is constrained to fit Flash.

## Paper boilerplate (you can cite/explain)
- Hardware: Arduino Nano 33 BLE Sense Rev2 (nRF52840, 64 MHz Cortex-M4F, 1 MB Flash, 256 KB RAM).
- Protocol: We report inference-only latency to ensure fairness across models with heterogeneous preprocessing pipelines (CNN requires INT8 packing; ML models require feature engineering). Each result is averaged over R runs after W warmups.
- Models:
  - 1D_CNN: TFLite Micro INT8; quantized inference via TFLM interpreter.
  - LightGBM/XGBoost: code-generated pure C predictors; on-device 20-dim handcrafted features.
  - ADANN: pure C FP32 head (FPU-accelerated). ADANN_LightGBM: simplified fusion on device.
- Metrics: we report mean latency and inferences-per-second (IPS). For very small times (XGBoost), we use microseconds for readability.

## Troubleshooting
- Flash overflow with hybrid models: reduce LightGBM `n_estimators` / `max_depth` in Arduino-mode training or disable fusion.
- CNN latency variance (~±0.2 ms): increase runs, add warmup, optionally perform multiple `Invoke()` per measurement iteration and average.
- Transformer not supported on Arduino due to missing `BATCH_MATMUL` in current TFLM build.
